## 8-Point Design Review Checklist

### 1. Visual Consistency

**Token Usage**: Excellent. All five components use design token-backed CSS custom properties (via Tailwind `text-on-surface`, `bg-background`, `text-error`, `bg-accent`, etc.) with no raw hex values. Colors, typography, spacing, border-radius, and shadows all trace back to `tokens.json` through `app.css` @theme variables.

**Issue found**: The Button `primary` variant uses `bg-primary` / `hover:bg-primary-light` which produces a dark-to-slightly-less-dark hover transition (slate-900 to slate-700). This is inconsistent with the design system guideline that primary CTA buttons should use `accent` color (blue-600), as demonstrated in HeroSection's inline button. The `primary` variant name suggests it is the default CTA action button, but it renders as a dark slate button rather than the blue accent CTA the design system intends.

### 2. Responsive Behavior

**SectionWrapper**: Correctly implements the Section Padding Strategy from the design system (`px-4 py-16` mobile, `md:px-6 md:py-20`, `lg:px-8 lg:py-24`). Typography scaling for h2 follows the documented scale (`text-2xl` -> `md:text-3xl` -> `lg:text-4xl`). Subtitle scales from `text-base` to `md:text-lg`. Max-width container at `max-w-7xl` is appropriate.

**Button, Card, Input, Textarea**: These are content-level components that correctly do not impose their own breakpoint-specific layout changes -- they inherit responsive context from their parent containers. This is the correct approach.

**Minor note**: Button text sizes are fixed per size prop (`text-sm`, `text-base`, `text-lg`) with no responsive scaling, which is acceptable for a button component.

### 3. Component Composition

Components are well-decomposed with clear single responsibilities. SectionWrapper provides consistent section layout with optional title/subtitle. Button uses variant and size maps for clean style composition. Input and Textarea share a near-identical pattern (label, field, error) which is a candidate for extracting a shared FormField wrapper, but acceptable at this stage.

Card accepts `className` for composition extensibility, which is good. However, Button does not accept `className`, limiting override capability for edge cases.

### 4. Accessibility

**Strengths**:
- SectionWrapper: Uses semantic `<section>` with `id` and `aria-label`
- Input/Textarea: Proper `<label htmlFor>`, `aria-invalid`, `aria-describedby` for error association, `role="alert"` on error messages, required indicator with `aria-hidden`
- Button: `min-h-11` (44px) meets touch target requirement, `focus-visible` ring present
- All interactive elements use `focus-visible` instead of `focus` (correct -- avoids focus ring on mouse click)

**Issues found**:
- Button: When `disabled={true}`, the button still receives pointer events because only visual styling (`opacity-50 cursor-not-allowed`) is applied but no `pointer-events-none` class. While native `disabled` attribute does prevent clicks, the cursor styling alone does not prevent hover state transitions, which creates visual noise on disabled buttons.
- Input/Textarea: The focus ring uses `focus-visible:ring-accent` (maps to `--color-accent: #2563eb`) while Button uses `focus-visible:ring-accent-focus` (maps to `--color-accent-focus: #3b82f6`). These are different blue shades. Focus ring color should be consistent across all interactive components -- the design system specifies `accent-focus` (#3b82f6) as the focus ring color.

### 5. Interaction Design

**Hover states**: Present on Button (color shift), Card (translateY lift + shadow-card-hover), Input/Textarea (border opacity increase). All correct per design system.

**Focus-visible states**: Present on all interactive elements with ring-2 and offset.

**Active state**: Button has `motion-safe:active:scale-[0.98]` per design system. Card does not have an active state, which is correct as it is a container, not an interactive element.

**Disabled state**: Button has `opacity-50 cursor-not-allowed`. Input/Textarea have `disabled:cursor-not-allowed disabled:opacity-50`. Consistent pattern.

**Transitions**: All use `motion-safe:` prefix respecting `prefers-reduced-motion`. Button and Card use `motion-safe:duration-200` (matching the `transition.normal` token of 200ms). Input/Textarea use `motion-safe:duration-200`. Consistent.

**Issue**: Card hover effect uses `hover:-translate-y-0.5` (2px) which is close to but not exactly the design system's specified `translateY(-2px)`. Tailwind's `-translate-y-0.5` maps to `-2px` (0.5 * 4px base = 2px), so this is actually correct.

### 6. Typography Hierarchy

SectionWrapper h2 uses `text-2xl font-bold leading-tight` with proper responsive scaling. Subtitle uses `text-base leading-relaxed text-on-surface-muted`. This follows the design system's typography guidelines.

Input/Textarea labels use `text-sm font-medium` which is appropriate for form labels. Error messages use `text-sm text-error`.

Button text inherits from size classes with `font-semibold`, which is correct for button text.

### 7. Spacing and Alignment

SectionWrapper: `mb-4 md:mb-6` between title and subtitle, `mb-10 md:mb-12` between subtitle and content. These use the 4px grid system consistently.

Input/Textarea: `mb-1` between label and field, `mt-1` between field and error. Form field internal padding `px-4 py-2.5` is comfortable.

Button padding scales with size: `px-4 py-2` (sm), `px-6 py-2.5` (md), `px-8 py-3` (lg). All on the 4px grid except `py-2.5` (10px) which is a half-step -- acceptable for vertical centering.

Card has no internal padding -- relies on children to provide content padding. This is a valid composition pattern but should be documented.

### 8. Platform Conventions

Web platform conventions are followed correctly:
- Semantic HTML elements (`section`, `button`, `input`, `textarea`, `label`)
- Mobile-first responsive approach (base styles are mobile, enhanced with `md:` and `lg:`)
- Touch targets meet 44px minimum (`min-h-11` on Button)
- Cursor utilities (`cursor-pointer`, `cursor-not-allowed`) applied appropriately
